Top = what almost never changes. Bottom = the concrete and executable.

SDD inverts the usual flow: the spec is the source of truth, code is generated to satisfy it. The constitution sits at the top of the hierarchy — the most stable layer. Each layer down is more concrete and changes more often.
Definition

The constitution

  • Non-negotiable rules in .specify/memory/constitution.md.
  • Every spec, plan and task must align with it.
  • Small: verifiable principles, not implementation detail.

Golden rule

It states what must always hold, not which technology to use.

Four things: non-negotiable, fixed path the agent reads, everything aligns to it, and it stays small. The golden rule is the whole talk in one line — WHAT must hold, never WHICH technology. The common failure is a giant constitution nobody follows.

Verifiable principle

Traceability

Every change must be traceable end to end:

Requirement → Specification → Contract → Task → Test → Code

A task without a spec is invalid.

A test without a business rule is incomplete.

Walk it left to right — and backwards: from any line of code you can ask "which requirement justifies this?". The two red cards are the enforcement rules: orphan tasks and orphan tests mean the chain is broken.

The key question

Where do React and Node.js go?

Below the constitution. The constitution only says: "use technologies approved by the stack" — never names one.

tech-stack.md · practical

# Technology Stack

## Frontend
- React, TypeScript, Vite

## Backend
- Node.js, NestJS

## Database
- PostgreSQL

ADR · professional

# ADR-001 · React

Status:  Accepted

Context:
A modern SPA is required.

Decision:
React is the primary framework.

Consequences:
- Large ecosystem
- Easy TS integration
Two options. tech-stack.md is a simple approved-tech list — one indirection so the constitution never needs editing when the stack changes. An ADR captures WHY, with a Status field that lets you supersede a decision without touching the constitution.
Anti-patterns

Common mistakes

Too long

40 principles nobody reads. If it doesn't fit on a page, it's a manual.

Tech in the constitution

Migrating the stack should never force a constitutional change.

Unverifiable rules

"Write clean code." You can't check a PR against it.

Silent changes

Editing a law with no reason, no migration. Governance must be explicit.

Four failure modes. Too long: it stops being read. Tech inside: couples laws to choices that should be free to change. Unverifiable: "clean/fast/good" can't gate a PR. Silent changes: amending without reason erodes the trust that makes it authoritative.
Real case · World Cup Pool

A constitution in practice

A World Cup prediction (betting pool) app — what must we protect? Fears become principles:

Generic engineering

Contract-First · Traceability · Domain Integrity · Separation of Concerns · Testability · Security by Design

Domain-specific

Prediction Immutability · Transparent Scoring · Fairness & Anti-Fraud · Deterministic Ranking · Auditability · Explicit Time Handling

React, Node, PostgreSQL, Redis → tech-stack.md. All 18 principles in full: examples/.specify/memory/constitution.md

Apply everything to a concrete product. For a betting pool the trust-killers are late or tampered predictions, points you can't explain, a non-reproducible leaderboard — those become principles. Prediction Immutability is the heart: after the deadline a prediction can never change, by anyone. None of them name a framework.
In practice

The Spec Kit flow

# Scaffold .specify/ with your agent
specify init my-world-cup-pool

Then the slash commands generate each artifact, in order:

/speckit.constitution→ /speckit.specify→ /speckit.plan→ /speckit.tasks→ /speckit.implement

Constitution first (the laws). Everything else is built — and checked — against it.

The order is the methodology. specify init scaffolds .specify/. Then, inside your agent, run the commands left to right: constitution, spec, plan, tasks, and only at the end the implementation. Each step is checked against the constitution.
