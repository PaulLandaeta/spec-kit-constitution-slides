Context

What is Spec-Driven Development?

The idea

Specifications don't serve the code.

The code serves the specification.

The shift

From "make the agent write code faster"…

…to "make the agent participate in engineering reliably": process, artifacts and constraints.

Spec Kit is GitHub's open-source toolkit for SDD: the specify CLI, templates and commands for your agent.

Traditional dev: specs are throwaway docs that "serve" the code. SDD inverts it: the spec is the source of truth and the code is generated to satisfy it. The point isn't speed — it's making the agent a reliable engineering participant by giving it process, artifacts and constraints. Spec Kit is GitHub's toolkit that ships the `specify` CLI plus templates and slash-commands for your coding agent.

Where the constitution fits

Top: what almost never changes. Bottom: the concrete and executable.

Read this top to bottom. The constitution is the most stable layer — it should barely change over the project's life. As you go down, each layer is more concrete and changes more often: standards, then architecture decisions, then per-feature specs, then the day-to-day tasks. A good test: a decision is at the right level if changing it doesn't force you to edit the layer above.
Definition

The constitution

  • The project's non-negotiable rules.
  • Lives in .specify/memory/constitution.md.
  • Specs, plans and tasks must align with it.
  • It is small: verifiable principles, not implementation instructions.

Golden rule

The constitution states what must always hold, not which technology to use.

Four things to remember: it's non-negotiable, it lives at a fixed path the agent reads, everything else must align to it, and it stays small. The golden rule is the whole talk in one line — it says WHAT must always hold, never WHICH technology. The most common failure mode is a giant constitution that becomes a manual nobody follows.
Content

What DOES go in the constitution

Verifiable principles

  • Contract-First Development
  • Separation of Concerns
  • Traceability
  • Testability

Boundaries and rules

  • Security by Design
  • Explicit Error Handling
  • No Hidden Behavior

Each one is a verifiable law: you can check whether a spec or a PR honors it.

These are principles, not technologies. The key property is "verifiable": for each one you can look at a spec or a pull request and answer yes/no — did it honor the rule? If a candidate rule can't be checked that way, it's probably a guideline or a preference, not a constitutional principle.
Example principle

Contract-First Development

# Project Constitution

## 1. Contract-First Development

Every public feature must be defined first by a verifiable
contract before it is implemented.

A contract may be:
- OpenAPI for REST endpoints.
- JSON Schema for data structures.
- Gherkin for acceptance criteria.
- Type definitions for internal interfaces.
- Preconditions and postconditions for business rules.

No endpoint, critical component or business rule may be
implemented without a prior contract.
Notice the shape of a good principle: a clear rule ("define a contract first"), the allowed forms a contract can take, and a hard boundary ("nothing ships without one"). It never says OpenAPI is mandatory over JSON Schema — that choice is left open. This is the level of abstraction you're aiming for in every principle.
Verifiable principle

Traceability

Every change must be traceable end to end:

Requirement → Specification → Contract → Task → Test → Code

A task without a related specification is invalid.

A test without a related business rule is incomplete.

Walk the chain left to right: a requirement becomes a spec, the spec defines a contract, the contract breaks into tasks, each task has a test, and only then code. The power is being able to walk it backwards too: from any line of code you can ask "which requirement justifies this?". The two red cards are the enforcement rules — orphan tasks and orphan tests are red flags that the chain is broken.
Content

What does NOT go in the constitution

Concrete technology

  • Use React
  • Use Node.js
  • Use MongoDB / PostgreSQL
  • Use Tailwind

Code conventions

  • Components in PascalCase
  • Use Zustand
  • Use Express

All of this belongs in auxiliary files → tech-stack.md, guidelines, ADRs.

None of these are wrong choices — they're just at the wrong level. The litmus test: if migrating from React to Angular, or Mongo to Postgres, would force you to edit the constitution, then that detail never belonged there. Push it down to tech-stack.md, guidelines, or an ADR.
Organization

Recommended .specify/ structure

.specify/ │ ├── memory/ │ ├── constitution.md # Project base laws │ ├── frontend-guidelines.md # UI, components, state │ ├── backend-guidelines.md # APIs, services, data │ ├── api-contracts.md # Endpoint contracts │ ├── testing-guidelines.md # Testing strategy │ └── security-guidelines.md # Cross-cutting security │ ├── tech-stack.md # React, Node, PostgreSQL… │ └── adr/ ├── adr-001-react.md ├── adr-002-nodejs.md └── adr-003-postgresql.md
This whole tree is in the repo under examples/.specify — you can copy it as a starting point. The two highlighted files are the ones agents read most: the constitution for the laws, and tech-stack.md for the approved technologies. Everything in memory/ is meant to be consulted by the agent on demand.

frontend-guidelines.md

Component Contract

# Frontend Guidelines

## Component Contract

Every reusable component must declare:
- Props
- Possible states
- Emitted events
- Error / loading / empty cases

Component: UserCard

Props:
- user: UserSummary
- onSelect?: (userId: string) => void
- variant?: "compact" | "detailed"

States:
- default
- loading
- empty
- error
A component contract forces you to think about every state up front — not just the happy path. Loading, empty and error are exactly the states developers forget. This lives in frontend-guidelines.md, applying the constitution's "No Hidden Behavior" and "Explicit Error Handling" to the UI layer.
api-contracts.md

Endpoint Contract

POST /users

Authentication: Required.
Authorization:  Only admin users.

Request Body:
{ "name": "string", "email": "string" }

Success Response:
201 Created

Error Responses:
400 Validation error
401 Unauthenticated
403 Unauthorized
409 Email already exists

Business Rules:
- Email must be unique.
- Name cannot be empty.
The contract declares every error before any code exists — this is "Error Handling by Contract" made concrete. Note the 409 for a duplicate email: that's a business rule surfaced as a specific status, not a generic 500. The agent can generate both the implementation and the tests from this.
backend-guidelines.md

Service Contract

Use Case: CreateUser

Preconditions:
- Email format is valid.
- Email does not already exist.

Postconditions:
- User is persisted.
- UserCreated event is emitted.
- Audit log is created.

Domain Errors:
- InvalidEmail
- EmailAlreadyExists
Preconditions and postconditions are the domain-layer version of a contract — pure business rules, independent of HTTP or the database. Domain errors get meaningful names (InvalidEmail, EmailAlreadyExists) instead of leaking framework exceptions. This is "Domain Integrity" and "Separation of Concerns" in practice.
The key question

Where do React and Node.js go?

At a level below the constitution. Two options:

Option 1 · tech-stack.md

The most practical. A list of approved technologies.

Option 2 · ADR

More professional. A documented decision with context and consequences.

↓ Go down to see both

This is the question everyone asks. The answer is never "in the constitution". Either keep a simple approved-tech list (tech-stack.md), or — for important, contested decisions — write an ADR that records the context and the trade-offs. Press down to compare both.
Option 1

tech-stack.md

# Technology Stack

## Frontend
- React
- TypeScript
- Vite

## Backend
- Node.js
- NestJS

## Database
- PostgreSQL

## API
- OpenAPI

And the constitution only says: "Every implementation must use technologies approved by the Technology Stack." It never mentions React directly.

The trick: the constitution points to this file by reference, never by name. So the law "use approved technologies" stays true forever, while the actual list evolves freely. One indirection buys you a constitution that never needs editing when the stack changes.
Option 2

adr-001-react.md

# ADR-001

Title:   React as Frontend Framework
Status:  Accepted

Context:
A modern SPA is required.

Decision:
React will be the primary framework.

Consequences:
- Large ecosystem
- Moderate learning curve
- Easy TypeScript integration

It lets you change decisions without touching the constitution.

An ADR captures WHY, not just what. Months later, when someone asks "why React and not Vue?", the context and consequences are right there. Crucially the Status field lets you supersede a decision (Accepted → Superseded by ADR-014) without rewriting history or touching the constitution.
Simple rule

So you don't get confused

constitution

laws that never change

guidelines

how we apply those laws per layer

contracts

verifiable agreements between parties

spec

what a concrete feature asks for

tasks

executable work

the common mistake

dumping everything into the constitution

If you remember nothing else, remember these five words and what each owns. The sixth card is the trap: when in doubt people cram everything into the constitution, and it rots into an unreadable rulebook. Keep each concern in its own file.
Anti-patterns

Common mistakes

Too long

40 principles nobody reads. If it doesn't fit on a page or two, it's not a constitution — it's a manual.

Tech in the constitution

"Use React / PostgreSQL / Tailwind." Migrating the stack should never force a constitutional change.

Unverifiable rules

"Write clean code." You can't check a PR against it. Every principle must be answerable yes/no.

Silent changes

Editing a law with no reason, no migration, no updated tests. Governance must be explicit.

Four failure modes I see again and again. Too long: it stops being read. Tech inside: it couples the laws to choices that should be free to change. Unverifiable: "clean", "fast", "good" can't gate a PR. Silent changes: amending a constitution without a documented reason and migration erodes the trust that makes it authoritative.
Real case

Constitution for a World Cup prediction pool

A World Cup prediction (betting pool) app. What matters most to protect?

  • Fairness — nobody predicts late or tampers
  • Locking — prediction immutable after the deadline
  • Reproducible scoring — you know why you got those points
  • Deterministic ranking — same facts, same leaderboard
  • Auditability — every critical action leaves a trace
  • Traceability — requirement → … → code

React, Node, PostgreSQL, Redis → go in tech-stack.md, not here.

Now apply everything to a concrete product. For a betting pool, the things that can kill trust are: late or tampered predictions, points you can't explain, and a leaderboard that isn't reproducible. Those fears become principles. Notice none of them mention a framework — the stack still lives in tech-stack.md.
World Cup Pool · 1–6

Foundational principles

1. Contract-First — nothing is built without a contract.

2. Traceability — Requirement → Spec → Contract → Task → Test → Code.

3. Domain Integrity — the domain doesn't depend on frameworks or DB.

4. Prediction Immutability — the prediction freezes after the deadline.

5. Transparent Scoring — no hidden scoring logic.

6. Fairness & Anti-Fraud — no late predictions or tampering.

Principles 1–3 are generic engineering laws; 4–6 are domain-specific to a prediction pool. Prediction Immutability is the heart of fairness: once the deadline passes, a prediction can never change — not by a user, an admin, a job, or an API call.
World Cup Pool · 7–12

Security, time and testing

7. Auditability — actor, action, timestamp, previous/new state.

8. Explicit Time Handling — explicit time zones, no ambiguity.

9. Error Handling by Contract — every expected error, declared.

10. Security by Design — validate input, secrets out of the code.

11. Privacy & Data Minimization — only the necessary data.

12. Testability — every business rule has tests.

Explicit Time Handling matters more than it looks for a World Cup app: deadlines and kickoff times span many time zones, so a single authoritative time standard prevents "my prediction locked an hour early" disputes. Auditability is what lets you defend every score after the fact.
World Cup Pool · 13–18

Determinism and governance

13. Deterministic Ranking — same facts, same leaderboard.

14. Administrative Accountability — auditable admins, no silent changes.

15. Separation of Concerns — no business rules in UI/controllers/DB.

16. Recoverability — scores recalculable from the facts.

17. Observability — logs/metrics/traces for what's critical.

18. Specification Governance — on conflict, the constitution wins.

📄 All 18 in full: examples/.specify/memory/constitution.md

Determinism and Recoverability work together: because scores are recalculable from source facts (predictions + results + rules), you can always rebuild the leaderboard and prove it's correct. Principle 18 is the meta-rule: when any feature request conflicts with the constitution, the constitution wins — and changing it requires explicit governance.
In practice

The Spec Kit flow

# 1. Initialize the project with your agent
specify init my-world-cup-pool

Then, inside the agent, the slash commands generate the artifacts:

/speckit.constitution→ /speckit.specify→ /speckit.plan→ /speckit.tasks→ /speckit.implement

/speckit.constitution creates/updates .specify/memory/constitution.md. Everything else is built aligned with it.

The order is the whole methodology. First `specify init` scaffolds .specify/. Then, inside your agent, you run the slash commands left to right: constitution first (the laws), then the feature spec, then the technical plan, then the task breakdown, and only at the end the implementation. Each step is checked against the constitution.
